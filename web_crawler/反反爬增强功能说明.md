# 反反爬增强功能说明

## 概述

爬虫工具已集成强大的反反爬功能，可以应对各种非登录限制的反爬机制。

## 支持的反反爬机制

### 1. User-Agent轮换 ✅

**问题**: 网站检测相同的User-Agent

**解决方案**:
- 内置50+个真实浏览器User-Agent
- 自动轮换，避免重复使用
- 支持桌面端和移动端混合

**代码示例**:
```python
from core.anti_detection import UserAgentRotator

ua_rotator = UserAgentRotator(use_mobile=True)
ua = ua_rotator.get_random()  # 获取随机UA
```

### 2. 完整请求头模拟 ✅

**问题**: 网站检测请求头不完整

**解决方案**:
- 模拟Chrome、Edge、Safari完整请求头
- 包含sec-ch-ua等现代浏览器特征
- 随机添加DNT、Referer等可选头部

**效果对比**:
```
优化前:
  User-Agent: Python-requests/2.28.1

优化后:
  User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)...
  Accept: text/html,application/xhtml+xml...
  Accept-Language: zh-CN,zh;q=0.9,en;q=0.8
  Accept-Encoding: gzip, deflate, br
  Cache-Control: max-age=0
  sec-ch-ua: "Not_A Brand";v="8", "Chromium";v="120"
  ...
```

### 3. 智能请求间隔 ✅

**问题**: 请求过快被识别为爬虫

**解决方案**:
- 自适应延迟：错误多则自动增加延迟
- 随机间隔：1-5秒随机延迟
- 指数退避：遇到429错误自动增加等待时间

**配置示例**:
```python
# 隐形模式（高隐蔽，慢速度）
fetcher = create_stealth_fetcher(
    concurrency=2,      # 低并发
    timeout=45
)

# 快速模式（平衡模式）
fetcher = create_balanced_fetcher(
    concurrency=3,      # 中等并发
    timeout=30
)
```

### 4. 智能重试机制 ✅

**问题**: 网络波动或临时封禁导致失败

**解决方案**:
- 根据错误类型采取不同策略
- 429错误：指数退避，最多重试5次
- 403错误：更换UA和代理，重试2次
- 503错误：等待5秒后重试
- 404错误：不重试，直接跳过

**重试策略表**:

| 错误码 | 重试次数 | 延迟策略 | 额外操作 |
|--------|----------|----------|----------|
| 429 (频率限制) | 5次 | 指数退避2^n秒 | 更换UA |
| 403 (拒绝访问) | 2次 | 2-5秒随机 | 更换UA+代理 |
| 503 (服务不可用) | 3次 | 5-10秒 | 等待后重试 |
| 500/502/504 | 3次 | 2^n秒 | 直接重试 |
| 404 | 1次 | 0秒 | 不重试 |

### 5. 代理IP支持 ✅

**问题**: IP被封或限制频率

**解决方案**:
- 支持HTTP/HTTPS代理
- 自动轮换代理
- 失败代理自动标记

**使用方法**:
```python
from core.anti_detection import AntiDetectionConfig

config = AntiDetectionConfig(
    use_proxy=True,
    proxy_list=[
        "http://proxy1.example.com:8080",
        "http://user:pass@proxy2.example.com:8080",
    ]
)
```

### 6. 429频率限制处理 ✅

**问题**: 请求过快返回429错误

**自动处理**:
```
[RATE LIMIT] 429 detected, waiting 3.5s before retry 1
[RATE LIMIT] 429 detected, waiting 5.2s before retry 2
[RATE LIMIT] 429 detected, waiting 8.1s before retry 3
```

### 7. 连接错误恢复 ✅

**问题**: 网络波动导致连接失败

**自动处理**:
- 连接超时自动重试
- DNS解析失败重试
- SSL证书错误处理
- 操作系统错误恢复

## 使用模式

### 模式1: 隐形模式 (推荐用于严格反爬网站)

```python
from core.enhanced_fetcher import create_stealth_fetcher

fetcher = create_stealth_fetcher(
    concurrency=2,    # 低并发，减少被封概率
    timeout=45        # 长超时，等待复杂页面
)

# 特点:
# - 延迟: 2-5秒
# - 并发: 2个
# - 重试: 最多5次
# - UA: 混合移动端
# - 速度: 较慢但成功率极高
```

**适用场景**:
- 有严格反爬机制的网站
- 需要极高成功率
- 可以接受较慢速度

### 模式2: 平衡模式 (推荐日常使用)

```python
from core.enhanced_fetcher import create_balanced_fetcher

fetcher = create_balanced_fetcher(
    concurrency=3,    # 中等并发
    timeout=30
)

# 特点:
# - 延迟: 0.5-2秒
# - 并发: 3个
# - 重试: 最多3次
# - UA: 桌面端
# - 速度: 平衡
```

**适用场景**:
- 一般反爬机制的网站
- 需要平衡速度和成功率
- 日常使用

### 模式3: 自定义模式

```python
from core.anti_detection import AntiDetectionConfig
from core.enhanced_fetcher import EnhancedAsyncFetcher

config = AntiDetectionConfig(
    use_proxy=True,
    proxy_list=["http://proxy1.com:8080"],
    rotate_ua=True,
    use_mobile_ua=True,
    adaptive_delay=True,
    min_delay=1.0,
    max_delay=3.0,
    max_retries=5
)

fetcher = EnhancedAsyncFetcher(
    anti_detection_config=config,
    concurrency=2,
    timeout=30
)
```

## 在GUI中启用反反爬

### 方法一: 配置文件

创建 `config.json`:
```json
{
  "anti_detection": {
    "mode": "stealth",      // stealth 或 balanced
    "concurrency": 2,
    "timeout": 45,
    "use_proxy": false,
    "proxy_list": []
  }
}
```

```bash
python enhanced_crawler.py \
  --urls urls.csv \
  --mode all \
  --anti-detection stealth \
  --concurrency 2 \
  --timeout 45
```

### 方法三: GUI设置界面

在"设置"标签页中:
- [x] 启用反反爬模式
- 模式选择: [隐形模式 ▼]
- 并发数: [2 ▼]
- 超时时间: [45 ▼] 秒

## 故障排查

### 问题1: 仍然被频繁封禁

**解决方案**:
1. 降低并发数到1-2
2. 增加延迟到3-8秒
3. 启用代理IP
4. 使用隐形模式

### 问题2: 速度太慢

**解决方案**:
1. 使用平衡模式
2. 减少超时时间
3. 筛选响应快的网站
4. 分批并行处理

### 问题3: 代理不工作

**检查事项**:
1. 代理格式是否正确: `http://host:port`
2. 是否需要认证
3. 代理是否可用
4. 网络是否通

## 最佳实践

1. **首次爬取**: 使用隐形模式，确保成功率
2. **日常爬取**: 使用平衡模式，平衡速度和成功率
3. **紧急爬取**: 增加并发，但注意被封风险
4. **定期更新**: 更新User-Agent池和代理列表
5. **监控日志**: 观察429/403频率，调整策略
